Fejlsøgnings-cases

Dokumentation af komplekse tekniske udfordringer, der er identificeret og løst systematisk på tværs af Pixel Wizards-projektet og H5/H6-forløbet. Hvert case følger strukturen: Problem → Analyse → Rodårsag → Løsning.

ProxmoxiLO / IPMILinux NetworkingLayer 2 Projekt: Pixel Wizards (H6)
Problem: Et automatiseringsscript kunne ikke tænde offline backup-serveren via iLO-interfacet. ipmitool fejlede med Error: Unable to establish IPMI v2 / RMCP+ session.
Fejlsøgning:
Trin 1: Netværksverifikation via nc -uvz <ilo-ip> 623 bekræftede at UDP port 623 var åben — L3/L4 firewall-blokering udelukket.
Trin 2: En curl-kommando mod iLO-interfacet returnerede No route to host — lokalt routing-problem på Proxmox-hosten.
Trin 3 – Rodårsag: ip neigh show viste at ARP-opslag fejlede med FAILED — isoleret til Layer 2. Gennemgang afslørede en konflikt: både en VLAN-aware bridge (vmbr2) og et manuelt subinterface (vmbr2.250) eksisterede, hvilket brød VLAN-tagningen.
Løsning: vmbr2.250 subinterface slettet. IP-adresse tildelt direkte til VLAN-aware bridge. Manglende VLAN-konfiguration på fysisk switch-port identificeret og rettet.
Resultat: IPMI-session etableres korrekt. Air-gap backup-scriptet kan tænde og slukke offline-serveren automatisk.

Windows ServerDHCP FailoverActive DirectoryPowerShell Projekt: Pixel Wizards (H6)
Problem: Nyudrullede Windows-klient-VM'er modtog APIPA-adresser (169.254.x.x) i stedet for gyldige DHCP-leases.
Fejlsøgning:
Trin 1: DHCP-serverloggen viste at ingen DHCP DISCOVER-pakker nåede frem til serverne — problemet var server-side.
Trin 2 – Rodårsag 1: Get-DhcpServerv4Binding afslørede at DHCP-servicen kun lyttede på management-interfacet — ikke på klient-VLAN-interfacet.
Trin 3 – Rodårsag 2 (dybere): Get-DhcpServerInDC viste at serverne var autoriseret i AD med gamle, forkerte IP-adresser. Den forkerte AD-autorisering forhindrede korrekt interface-binding. 
# Kommando der afslørede fejlen:
Get-DhcpServerInDC
# Output viste gamle IP-adresser fra tidligere netværkssegment
Løsning:
  1. Forældede DHCP-objekter fjernet: Remove-DhcpServerInDC
  2. Re-autoriseret med alle korrekte IP-adresser: Add-DhcpServerInDC
  3. DHCP-service genstartet — bandt nu korrekt til begge interfaces.
Resultat: DHCP Failover fungerer korrekt. Fuld redundans verificeret.

AnsibleProxmox APIPythonYAML Projekt: H5 + Pixel Wizards (H6)
Problem: Playbooks fejlede konsekvent med 401 Unauthorized mod Proxmox API, selv med gyldigt token verificeret via curl.
Kæde af overlappende fejl:
Fejl 1 – Privilege Separation: API-tokenet var oprettet med "Privilege Separation" aktiveret — forhindrede token i at arve root@pam rettigheder.
Fejl 2 – Token Format: community.proxmox-modulet forventer det korte token-navn, ikke det fulde ID. Modulet bygger selv det fulde ID internt. 
# FORKERT – 401:
api_token_id: "root@pam!ansible2"

# KORREKT:
api_token_id: "ansible2"
Fejl 3 – Variabelnavne: Playbooks og templates brugte forskellige navne for samme secret, hvilket resulterede i tomme variabler ved runtime.
Fejl 4 – vars_prompt: En vars_prompt-sektion tvang manuel indtastning med hardcoded gammelt token som default — overskrev korrekte værdier fra group_vars/all.yml.
Fejl 5 – Inventory script: dynamic_inventory.py manglede import os og havde hardcoded gammelt API secret som fallback.
Løsning: Alle fem punkter rettet systematisk. Playbook kørte derefter fejlfrit.
Resultat: Ansible deployede succesfuldt 4 Windows VM'er via Proxmox API — klonet, tagget og startet automatisk.

WSL2Hyper-VVPN / ICSWindows Networking Projekt: Ansible Project Generator (H5)
Problem: WSL2 fik APIPA-lignende IP og kunne ikke tilgå interne VPN-ressourcer eller resolvere interne DNS-navne.
Fejlsøgning:
Trin 1: Midlertidig offentlig DNS i /etc/resolv.conf bekræftede at Internetforbindelsen i WSL var intakt — problemet var specifikt VPN-routing.
Trin 2: En eksisterende ~/.wsl-config i Linux-hjemmemappen identificeret som irrelevant (Dell-systemværktøj).
Rodårsag: ICS var konfigureret til at dele Ethernet-adapteren — ikke VPN-adapteren. WSL fik internet, men ingen adgang til det interne VPN-netværk.
Løsning: ICS deaktiveret på Ethernet-adapter, aktiveret på VPN-adapter med vEthernet (Default Switch) som destination. WSL genstartet. 
# C:\Users\<bruger>\.wslconfig
[wsl2]
networkingMode=bridged
vmSwitch=Default Switch
Resultat: WSL2 tilgår fuldt ud både internet og interne VPN-ressourcer inkl. Proxmox API og intern DNS.

Proxmox VECorosyncCluster NetworkingFirewall Projekt: Pixel Wizards (H6)
Problem: pvecm status viste Quorate: No. Ingen VM'er, API-tokens eller ændringer kunne oprettes — alt fejlede med no quorum!.
Fejlsøgning:
Trin 1: Ping-test mellem alle noder — fuld ICMP-konnektivitet bekræftet. Problemet var ikke grundlæggende netværksforbindelse.
Rodårsag: Corosync kommunikerer via UDP port 5404 og 5405. En ekstern firewall blokerede disse porte mens normal ICMP og TCP gik igennem.
Løsning: UDP port 5404 og 5405 åbnet i den fysiske netværksfirewall for Proxmox-nodernes Corosync-IP-adresser. Quorum genetableret umiddelbart.
Resultat: Cluster quorum genetableret. Alle cluster-operationer og API-token-oprettelse fungerer.

CephProxmox VEOSDStorage Cluster Projekt: Pixel Wizards (H6)
Problem: Ceph-monitoren rapporterede en eskalerende mængde "slow ops" — fra 39 til 437 på under 3 minutter. Monitor-rollen skiftede gentagne gange: leader → electing → probing.
Loganalyse: 
15:30:01 - reporting 39 slow ops   (oldest: osd_failure osd.14)
15:31:56 - reporting 232 slow ops
15:32:11 - reporting 437 slow ops  → monitor skifter til 'electing'
15:32:06 - reporting 362 slow ops  → 'electing'
15:33:06 - reporting 1 slow ops    (osd_alive: want up_thru 854)
Analyse: OSD.14 fejlede og svarede ikke inden for timeout. Dette udløste en kædereaktion: alle øvrige OSD'er sendte simultant osd_failure-rapporter til monitoren, som overbelastede monitor-processen og resulterede i gentagne leder-valg.
Rodårsag: OSD.14 gik offline. Den hurtige eskalering af slow ops indikerede yderligere at Ceph-monitor-netværket og storage-netværket ikke var tilstrækkeligt adskilt, hvilket forstærkede effekten.
Håndtering: OSD markeret out — Ceph redistributerede data til de resterende OSD'er. Storage-netværk verificeret. Clusteret genvandt HEALTH_OK efter re-distribution.
Resultat: Cluster restabiliseret. Data-integritet opretholdt via replikering. Hændelsen dokumenterede vigtigheden af et dedikeret Ceph-netværk adskilt fra Corosync.

Proxmox VELVM StorageHA ClusterLog Forensics Projekt: Pixel Wizards (H6) – 11. februar 2026
Problem: Domain Controller VM'en fejlede totalt — LVM disk-volumes var forsvundet. Fejl: no such logical volume pve/vm-disk-0. Ansible-automation (kørende kl. 02:00–04:00) blev mistænkt som årsag.
Forensisk tidslinje: 
02:56 - 03:17   Ansible kører: kloner template til 4 nye VMs. ALLE operationer OK.
04:00 - 10:00   4 succesfulde backup-jobs kører. System stabilt.
10:49 - 10:50   SSH root-login fra ekstern host (~1 minuts session)
11:05           VM stoppes manuelt via HA cluster resource manager
~11:30-12:30    LVM disk-volumes forsvinder
12:38           Første fejl: "no such logical volume"
12:38 - 13:00   12+ fejlede genstartsforsøg
~15:00          Laszlo gendanner VM fra PBS-backup
Bevis 1 – Ansible var IKKE årsagen: 9 timers og 21 minutters forskel mellem Ansible-afslutning (03:17) og første fejl (12:38). 4 succesfulde backup-jobs i mellemtiden.
Bevis 2 – Systematisk backup-udelukkelse: Gennemgang af alle 6 backup-jobs viste at specifikke VM'er var fraværende fra samtlige backup-jobs — fjernelsen skete inden Ansible-arbejdet.
Rodårsag (LVM): LVM logical volumes forsvinder kun ved: lvremove-kommando, VG-korruption, eller konfigurationsændring. Da kun denne ene VM var påvirket peger analysen mod manuel sletning.
Gendannelse: VM gendannet fra PBS-backup til ny VMID på Ceph-storage: 
qmrestore online-backup:backup/vm/<id>/<timestamp> <new-vmid> --storage CephPool-Pixel
# restore image complete (75 GB genoprettet med intakt data)
Konklusion: Den tekniske analyse udelukker Ansible-automatiseringen som årsag. Tidsgabet, de 4 succesfulde backup-jobs og den systematiske backup-udelukkelse dokumenterer at fejlen ikke opstod som følge af mit arbejde.

Ansiblecommunity.proxmoxPythonWSL2 Projekt: Ansible Project Generator (H5)
Problem: ansible-inventory -i inventory.yml fejlede med inventory source could not be verified by inventory plugin — selv mens curl mod samme endpoint fungerede perfekt.
Systematisk debugging:
Test 1 – Python proxmoxer direkte: Et test-script bekræftede at proxmoxer-biblioteket kommunikerede korrekt med Proxmox API — problemet var specifikt i Ansible inventory plugin-laget.
Test 2 – Minimal inventory: Inventory uden compose/filter-blokke fungerede — fejlen opstod i de avancerede blokke der forsøgte at hente guest agent-data, som ikke var tilgængelig på alle VMs.
Rodårsag: community.proxmox-pluginnen håndterede ikke gracefully at guest-agent-interfacet er inaktivt på visse VMs — returnerede fejl i stedet for null/tom værdi.
Løsning: Inventory-plugin erstattet med et custom dynamic_inventory.py Python-script med fuld kontrol over null-håndtering og fejltolerance.
Resultat: Dynamic inventory fungerer stabilt på tværs af alle VM-typer uanset guest agent-status.

AnsibleYAMLvars_prompt Projekt: Ansible Project Generator (H5)
Problem: generate_project.yml fejlede med multiple tilsyneladende uafhængige fejl der dukkede op ét ad gangen.
Fejlene i rækkefølge:
Fejl 1: Invalid vars_prompt data structure, found unsupported key 'regex' — Ansible-versionen understøttede ikke regex: og confirm: i vars_prompt. Udkommenteret.
Fejl 2: 'vm_user_1' is undefined — Variabelreference i prompt-teksten på et tidspunkt hvor variablen ikke eksisterede endnu. Reference fjernet fra teksten.
Fejl 3: Invalid variable file contents — YAML-syntaksfejl efter manuel redigering. En bindestreg (-) foran en udkommenteret blok brød YAML-parsingen. Hele blokken inkl. listen-elementet skal udkommenteres samlet.
Løsning: Sekventiel rettelse: kør playbook → ret fejl → kør igen. YAML-fejl er chainede.
Resultat: Playbook kører fejlfrit og starter interaktive prompts korrekt.

Windows ServerDFS-RActive DirectoryPowerShell Projekt: CloudFlix (H5) + Pixel Wizards (H6)
Problem: DFS Replikering synkroniserede ikke data mellem filserverne. Filer oprettet på én server dukkede ikke op på den anden.
Fejlsøgning:
Trin 1: Get-DfsrState og Event Viewer bekræftede at replikeringsgruppen var konfigureret korrekt men health-status viste fejl.
Rodårsag: Replikeringsgruppen var ikke korrekt initialiseret. Den første synkronisering kræver at én server markeres som "primary member". Dette trin var overset i det automatiserede setup-script.
Løsning: Replikeringsgruppe re-konfigureret med eksplicit primary member. Initial synkronisering startet manuelt. ABE verificeret aktivt på namespace.
Resultat: DFS replikering fungerer. Namespace tilgængeligt og transparent for brugere på begge filservere.

Testrapporter

Udvalgte testcases udført for at validere systemets funktionalitet og sikkerhed mod de opstillede krav.

FirewallVLAN Segmentering
Formål: Verificere at backup-serverne i VLAN 40 kun er tilgængelige fra specificerede administrations-VLANs.

Test Trin:
  1. Positiv Test: ping og HTTPS mod backup-serveren fra klient-VLAN og server-VLAN.
  2. Negativ Test: Samme tests fra gæste-VLAN — forventet timeout.
Resultat: Positiv test lykkedes. Negativ test timede ud. Firewall-politik fungerer korrekt.

PBSiLO / IPMIBash Scripting
Formål: Verificere den fuldautomatiserede Air-Gap synkroniseringsproces fra online til offline backup-server.

Test Trin:
  1. Hardware-verifikation: Login på iLO for at bekræfte servermodel, hostname og IP-adresse.
  2. Funktionel Test: Manuel eksekvering af sync_online_til_offline_backup.sh på online-serveren.
Forventet Resultat: Scriptet tænder offline-serveren via IPMI, synkroniserer snapshots og slukker serveren igen.

WinRMOpenSSHAnsiblePowerShell
Formål: Verificere at klienter fra hærdet template er korrekt forberedt til Ansible-fjernadministration.

Test Trin:
  1. Bloatware-fjernelse (K-49): Get-AppxPackage -Name "*Xbox*" skal returnere tomt resultat.
  2. UAC Remote Policy (K-48.3): LocalAccountTokenFilterPolicy skal være sat til 1.
  3. Port-verifikation (K-48): Test-NetConnection bekræfter TCP 5985 (WinRM) og TCP 22 (SSH) er åbne.
Resultat: Alle kommandoer returnerede korrekte værdier. Template klargør klienter korrekt til Ansible.

DHCPFailoverPowerShell
Formål: Verificere at DHCP-failover fungerer og klienter modtager IP selv når én server er nede.

Test Trin:
  1. Verificer aktiv failover: Get-DhcpServerv4Failover
  2. Stop DHCP-service på primær server: Stop-Service dhcpserver
  3. Forny IP-lease på klient: ipconfig /renew
  4. Verificer lease fra sekundær server. Start primær og verificer re-synk.
Resultat: Klienten modtog IP-adresse fra sekundær DC inden for failover-timeout. Redundansen verificeret.

AnsibleProxmox APIWinRM
Formål: Verificere at den komplette Ansible deployment-pipeline fungerer fra template-kloning til konfigureret VM.

Test Trin:
  1. Kør generate_project.yml og angiv VM-parametre via interaktive prompts.
  2. Kør create_control_nodes.yml — verificer VM'er klones og startes i Proxmox.
  3. Verificer VM-tags er sat korrekt (windows, control-node).
  4. Kør configure_control_nodes.yml — verificer WSL-installation via WinRM.
  5. Test Ansible-ping mod nyudrullet VM — forventet: pong.
Resultat: VM'er klonet, startet, tagget og konfigureret fuldt automatisk. Ping returnerede pong.
