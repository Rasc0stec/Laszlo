Air-Gap Backup Automation

  1. Home

Projektoversigt & Formål

Air-Gap Backup-projektet, udviklet som en del af Pixel Wizards (H6) infrastrukturen, løser en kritisk udfordring i enterprise backup-strategier: hvordan sikrer man offline backup-redundans uden at kompromittere sikkerhed eller introducere manuel overhead?

Løsningen er en fuldautomatiseret Air-Gap backup-proces der via IPMI/BMC power management kontrollerer en fysisk isoleret backup-server. Serveren tændes automatisk via netværket, synkroniserer 32 VM snapshots (12+ GiB dagligt) fra online Proxmox Backup Server til offline PBS, og slukkes igen - alt sammen uden manuel håndtering.

Resultatet er en backup-strategi der kombinerer convenience af cloud-backup med den fysiske sikkerhed af tape-backup: offline serveren er kun tilgængelig i gennemsnitligt 16 minutter dagligt, hvilket gør den immune over for ransomware, netværksangreb og utilsigtet sletning.

Implementation & Udfordringer

Bash Automation Script

Scriptet håndterer hele backup-cyklussen med omfattende error handling:

#!/bin/bash
# /usr/local/bin/sync_online_til_offline_backup.sh

OFFLINE_IP="10.10.250.212"
OFFLINE_PBS="10.10.40.212"
IPMI_USER="Administrator"
IPMI_PASS="<secret>"

# 1. Tænd server via IPMI
ipmitool -I lanplus -H $OFFLINE_IP -U $IPMI_USER -P $IPMI_PASS chassis power on

# 2. Vent på server boot (~2m 50s)
until ping -c1 $OFFLINE_PBS &>/dev/null; do sleep 5; done
sleep 60  # Vent på PBS services

# 3. Synkroniser alle datastores
proxmox-backup-manager sync-job run --store offline-backup

# 4. Sluk server ved success
ipmitool -I lanplus -H $OFFLINE_IP -U $IPMI_USER -P $IPMI_PASS chassis power off

Kritiske Fejlsøgningscases

Case #1: IPMI Connection Failure (Layer 2)
ipmitool fejlede med "Unable to establish RMCP+ session" selvom port 623 var åben. Rodårsag: ARP resolution fejlede grundet konflikt mellem VLAN-aware bridge (vmbr2) og manuel subinterface (vmbr2.250). Løsning: Fjernet subinterface, konfigureret VLAN direkte på bridge + rettet missing VLAN-config på fysisk switch-port.
Challenge: Timing & Reliability
Serveren skal være 100% stabil tændt under sync (12+ GiB transfer). Implemented ping-loop med timeout, service wait-periode, og error logging der efterlader serveren tændt ved fejl så manuel troubleshooting er mulig.

Læring & Takeaways

  • IPMI er kraftfuldt men finurligt: Layer 2 networking (ARP, VLAN) skal være 100% korrekt - IPMI tolererer ikke misconfig
  • Bash scripting kræver defensive practices: Altid håndter timeouts, log alle states, og design for fejl-modes
  • Air-Gap er en trade-off: Maksimal sikkerhed vs. convenience - automation mindsker manual overhead betydeligt
  • Real-world troubleshooting: Debugging på tværs af Proxmox, BMC firmware, switch-config og Linux networking gav dyb systemforståelse
